# 自研专利检索服务需求说明

## 1. 项目定位

本项目建设一套基于公司自有 OpenSearch `patent_index` 的专利检索后端服务，用于逐步替代当前 SaaS 平台依赖的外采专利检索服务。

服务对上提供 HTTP API，供 SaaS 平台或业务后端调用；对下连接公司自有 OpenSearch 专利索引。第一版目标是先完成可联调、可验证、可灰度替换的后端检索能力。

## 2. 替换对象

本项目第一轮对齐外采服务中的以下能力：

| 外采服务能力 | 自研服务要求 | 说明 |
|---|---|---|
| `patent_search` | 必须支持 | 核心专利检索接口 |
| `patent_get_detail` | 必须支持 | 专利详情查询接口 |
| `patent_get_citations` | 必须支持 | 引证/相关文献查询接口 |
| 查询参数 | 优先兼容 | 兼容 `q`、`ds`、`sort`、`page`、`page_size`、`highlight` |
| 返回字段 | 优先兼容 | 保持 SaaS 已消费字段的名称和含义 |

## 3. 兼容原则

本项目采用“对外兼容，内部自研”的原则。

对外兼容：

1. 保留外采服务核心参数命名：`q`、`ds`、`sort`、`page`、`page_size`、`highlight`。
2. 保留 SaaS 已依赖的核心返回字段：`patent_id`、`applicationNumber`、`documentNumber`、`title`、`abstract`、`applicant`、`currentAssignee`、`mainIpc`、`ipcMainList`、`applicationDate`、`documentDate`、`legalStatus`、`type`、`total`。
3. 业务接口成功响应优先保持与原外采服务接近的返回结构，不强制包裹统一响应体。
4. 返回字段以 SaaS 实际消费字段为优先，不要求外采服务字段 100% 全量保留。
5. 优先实现“替换接口地址后即可联调”的效果。
6. 不要求检索结果、排序结果、高亮结果与外采服务完全一致。

内部自研：

1. 查询语法解析由自研服务实现。
2. OpenSearch DSL 构造由自研服务实现。
3. 字段映射由自研服务实现。
4. 结果格式化由自研服务实现。
5. 参数校验、异常处理、日志、分页、排序由自研服务统一控制。

## 4. 第一版接口范围

第一版必须建设以下接口：

```text
POST /api/patent/search
GET  /api/patent/detail/{patent_id}
GET  /api/patent/citations/{patent_id}
GET  /health
```

第一优先级是：

```text
POST /api/patent/search
```

## 5. 第一版必须支持的能力

| 能力 | 第一版要求 |
|---|---|
| 标题检索 | 支持 |
| 摘要检索 | 支持 |
| 全文检索 `tscd` | 支持标题、摘要、权利要求、说明书 |
| IPC 检索 | 支持主 IPC 和 IPC 列表 |
| 申请人检索 | 支持 |
| 当前权利人检索 | 支持 |
| 申请日范围 | 支持 |
| 公开年范围 | 支持 |
| 法律状态 | 支持基础映射 |
| 专利类型 | 支持 |
| 分页 | 支持 |
| 排序 | 支持 `relation`、`!applicationDate` |
| 数据范围 | 支持 `ds=cn/all` |
| 高亮参数 | 接收 `highlight=0/1`，第一版允许基础支持或兼容保留 |

## 6. 第一轮不做范围

第一轮不做以下内容：

1. 不做前端页面。
2. 不做 SaaS 业务流程改造。
3. 不做专利数据入库、清洗、同步任务。
4. 不做报告生成。
5. 不做 AI 分析能力。
6. 不做专利图片/PDF 文件服务。
7. 不做复杂权限计费体系。
8. 不要求完全复刻外采服务召回结果和排序结果。
9. 不一次性支持所有复杂查询语法。

## 7. 阶段推进顺序

```text
阶段一：需求边界确认
阶段二：部署环境前置确认
阶段三：字段映射与接口设计
阶段四：项目骨架搭建
阶段五：最小检索链路跑通
阶段六：核心查询能力补齐
阶段七：详情与引证能力补齐
阶段八：接口兼容与异常处理完善
阶段九：测试与外采服务对比
阶段十：SaaS 联调与灰度验证
阶段十一：部署上线与交付沉淀
阶段十二：后续迭代优化
```

## 8. 当前项目状态

截至当前文档整理时，项目处于阶段二初步确认、阶段三收口前。

当前允许推进的工作：

1. 补齐部署环境剩余生产化问题。
2. 冻结字段映射、API 结构、查询语法范围。
3. 准备阶段四工程骨架任务清单。

当前不允许推进的工作：

1. 不直接开发复杂查询语法。
2. 不直接做 SaaS 联调。
3. 不直接进入性能优化。
4. 不在字段映射未冻结前实现完整业务检索。
5. 不在前三阶段验收前正式进入阶段四开发。

## 9. 第一版验收标准

第一版完成后至少满足：

1. 服务可启动。
2. `/health` 正常返回。
3. `POST /api/patent/search` 可查询真实 OpenSearch 数据。
4. `GET /api/patent/detail/{patent_id}` 可返回专利详情。
5. `GET /api/patent/citations/{patent_id}` 可返回引证/相关文献。
6. 业务接口成功响应与原外采服务返回结构接近。
7. 搜索返回包含 `total`、`page`、`page_size`、`records`。
8. 核心返回字段与 `docs/field_mapping.md` 一致。
9. 参数错误、查询语法错误、OpenSearch 异常有明确错误返回。
10. README、接口文档、查询语法文档、部署说明、测试报告齐全。
